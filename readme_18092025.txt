readme_19092025 (pilot “Siano AI”)

Versione: 19/09/2025

COS’È

“Siano AI” è una homepage comunale “viva” che:

racconta Siano (storia, galleria, festività);

elenca gli esercenti aderenti con pagine dedicate (QR);

integra un assistente AI che risponde usando solo dati locali (JSON).

È un pilot: low-cost (static hosting), modulare, facilmente estendibile.

TECNOLOGIE

Vite + React + TypeScript

Tailwind utility classes (classi inline, nessun config custom)

Framer Motion (transizioni/AnimatePresence)

lucide-react (icone)

Netlify (hosting + Functions per endpoint AI /.netlify/functions/ask)

Dati statici JSON in /public/data (home.json, venues.json, <slug>.json)

Facoltativo: VITE_AI_ENDPOINT per puntare a un endpoint AI esterno

AI upstream attuale: Hugging Face (router Chat Completions).
Chiave: variabile di ambiente HF_TOKEN su Netlify.

FILE/DATI CHIAVE
/public/data/home.json (Comune; struttura diversa dagli esercenti)

cityName, logoUrl, heroTitle, heroSubtitle, heroImages

about, festivities, project, pilot, gallery

openData (link JSON/CSV esercenti)

social, footer, theme (accent cobalto)

assistant: enabled, label, panelTitle, slugForQA (“home”), initialMessage

chat:

quickReplies (bottoni rapidi)

faq (Q/A fisse)

ctas (array di link, es. Sito, Open Data) → mostrate nella chat

Nota: la home NON ha config.phone/whatsapp/mapUrl. I pulsanti “Chiama / Indicazioni / WhatsApp” compaiono solo se definiti esplicitamente in chat.ctas.

/public/data/<slug>.json (Esercente)

config: name, hours, address, phone, whatsapp, mapUrl, assistantLabel

story: descrizione

menu (inteso come “catalogo/servizi”, non solo ristorazione):

specials (opz.)

categories[].items[] con price/tags/fav

chat: quickReplies, faq, ctas (opzionali)

Se mancano chat.ctas, la chat ricava automaticamente CTA da config (Chiama/WhatsApp/Indicazioni).

COMPORTAMENTO CHAT (ONLINE/OFFLINE)
Online (netlify/functions/ask.js)

Il widget chiama /.netlify/functions/ask.

La Function rileva profilo COMUNE vs ESERCENTE, compatta i dati JSON e istruisce il modello a usare SOLO i dati forniti.
Se un’informazione non esiste nel JSON → “Non disponibile”.

Chiude con CTA coerenti:
Comune: Sito/Open Data • Esercente: Chiama/WhatsApp/Indicazioni (o quelle in chat.ctas).

Hardening anti-errori:

Throttle per-IP (token bucket) per smorzare picchi;

Auto-retry su 429/5xx (con jitter);

Timeout hard al fetch dell’AI;

Cache in-memory dei JSON per 60s;

Mappatura errori: niente 500 “nudi”; restituisce 429 (rate limit) o 503 (indisponibile).
Il widget, su !ok, ricade nel fallback locale (UI non si blocca).

Offline (ChatWidget.tsx + ai/ask.ts)

Se l’endpoint non risponde, entra il fallback locale.

Riconosce intent semplici (orari, indirizzo, contatti, storia/about, progetto, eventi, open data) e suggerisce categorie del catalogo (“menu” ≡ “catalogo/servizi”).

Per saluti/domande generiche risponde con prompt di navigazione
(es. “Prova con: Storia di Siano • Progetto AI • Esercenti aderenti”), senza inventare contenuti.

Stile: bottoni/CTA in outline cobalto (pills) + bottone accent per WhatsApp; FAB, tab attivi e badge usano --accent (cobalto del tema).

COMPONENTI
HomePage.tsx

Carica home.json e venues.json.

Applica tema (blu cobalto), favicon, titolo.

Tabs: “Siano”, “Progetto pilota”, “Esercenti”. Scroll stabile (header sticky misura altezza e imposta scroll-padding-top).

Galleria a card con hover dolce.

Festività: mostra 3 elementi; toggle “Mostra tutte/meno”; shuffle deterministico singolo se ≥6.

Monta ChatWidget con slug="home".

SianoVenue.tsx (CUORE ESERCENTI)

Prende lo slug (es. /il-pirata), carica /public/data/<slug>.json.

Render intestazione (logo/nome), story/descrizione, catalogo (categorie+items).

CTA “Chiama/WhatsApp/Indicazioni” da config o chat.ctas; stile coerente con ChatWidget.

Monta ChatWidget con lo stesso slug.

ChatWidget.tsx

Carica /data/<slug>.json in client.

Prova prima l’endpoint AI; se fallisce, fallback locale (topic/faq/orari/indirizzo/servizi).

Autoscroll automatico all’ultimo messaggio (user o assistant).

CTA precedence: props.ctas → json.chat.ctas → (derivate da config) → [].

Comune (slug="home"): mostra solo chat.ctas (Sito, Open Data).
Esercente: se chat.ctas mancano, genera CTA da config.

Uniforma colori a tema cobalto; quick replies come chips.

netlify/functions/ask.js

Endpoint AI “online”.

Compatta i dati per COMUNE/ESERCENTE, costruisce il prompt, temperatura 0.2, max 90 parole.

Vietato inventare fonti/contatti.

Hardening (come sopra): throttle, retry, timeout, cache JSON 60s, mapping errori 429/503.

In caso di errore HTTP, il widget ricade sul fallback locale (nessun blocco dell’UI).

TROUBLESHOOTING RAPIDO

I pulsanti “Chiama/Indicazioni/WhatsApp” compaiono sulla home?
→ Verificare: la home non ha config.phone/whatsapp/mapUrl. Usare chat.ctas esplicite (es. {type:"link", label:"Sito", href:"…"}).

La chat risponde “Non disponibile” su domande fuori dato (es. popolazione)?
→ Corretto: la Function forza “solo dati locali”. Aggiungere i dati nel JSON o valutare in futuro il flag ibrido.

Vedi 500 nello Network?
→ Con l’hardening, la Function mappa gli errori in 429/503. Se compaiono 429/503 durante test pubblici: è rate-limit o upstream lento. Soluzioni:

attendere e riprovare (il widget usa fallback);

alzare tier HF (Inference Endpoint/Space dedicata);

verificare i logs Netlify (cold start, timeouts).

Stile pulsanti diverso tra home ed esercenti?
→ Assicurarsi che le var CSS di tema siano caricate e che il ChatWidget usi outline cobalto (già in codice).
